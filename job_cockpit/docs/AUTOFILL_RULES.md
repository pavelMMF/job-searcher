# Autofill and Auto-Apply Rules

These rules apply to every browser-driven application attempt in Job Cockpit.

## Goal

The agent should reduce typing and copying. It may fill known fields, attach the selected resume, prepare the page, and click final submit only when the form is fully recognized and safe.

## Hard Stops

The agent must stop and require human handoff for:

1. CAPTCHA, hCaptcha, reCAPTCHA, Cloudflare challenges, login checkpoints, MFA, OTP, password prompts, or anti-bot screens.
2. Broad legal acknowledgements, terms/account creation, background-check acknowledgement, arbitration, marketing/newsletter consent, or similar legal commitments.
3. EEO, demographic, disability, veteran, race, ethnicity, gender, sexual-orientation, pronoun, or similar fields unless a matching explicit value is present in `application_answers.json`.
4. Unknown required fields.
5. Multiple final submit candidates or ambiguous continue/review flows.
6. Work authorization or sponsorship yes/no questions unless explicit `application_answers` values are present.

## Answer Bank

Safe answers live in `job_cockpit/config/application_answers.json`.

Use `job_cockpit/config/application_answers.sample.json` as the template. Empty strings and `null` values mean "do not answer automatically".

Critical boolean fields:

- `work_authorized`: answer only if the relevant jurisdiction is clear and true for the vacancy.
- `requires_sponsorship`: answer only when the sponsorship answer is true or false for the vacancy.
- `remote_ok`, `willing_remote`, `willing_to_relocate`: used for safe remote/relocation choice fields.
- `eeo_race`, `eeo_gender`, `eeo_veteran_status`, `eeo_disability_status`: used only for matching EEO/demographic fields when explicitly configured.
- `legal_consent_application_processing`, `legal_consent_privacy`: may check narrow consent boxes for privacy notice, GDPR/data processing, and application-data handling only.

Do not infer `work_authorized` from a generic text field such as `Belarus, Georgia`.

## Allowed Automation

The agent may fill:

- name, email, phone, LinkedIn, GitHub, portfolio;
- current title/company when sourced from the master resume or answer bank;
- city, country, postal code, timezone only when explicitly configured;
- cover letter generated by Job Cockpit;
- salary, notice period, years of experience only when explicitly configured;
- safe remote/relocation checkboxes and radio buttons when explicit boolean values exist;
- EEO/demographic fields only when a matching explicit answer exists in `application_answers.json`;
- narrow privacy/application-processing consent only when explicitly enabled in `application_answers.json`;
- skill rating fields using conservative values already supported by the resume.
- safe screening questions via LLM, limited to real resume facts, role motivation, already-present tools, profile links, and explicit answer-bank facts.

The agent may attach the generated tailored resume or fallback resume.

## Audit

Every run writes:

- JSON report to `output/automation_runs`;
- screenshot beside the JSON when Playwright can capture it;
- `AUDIT_PATH` in the queue logs;
- `Audit JSON` link in the Auto Applies UI.

The audit must include metrics, submit readiness, manual fields, final URL, screenshot path, and safety rules.

## Submit Policy

Auto-submit is allowed only when:

- `auto_submit_allowed` is enabled in workflow settings;
- CAPTCHA/security detection is clear;
- all required fields are filled;
- no blocked legal/demographic/security field is present;
- any EEO or narrow application-processing consent fields were filled from explicit answer-bank values;
- exactly one final submit button is detected;
- the button is a final submit/send action, not a next/continue/review step.

If any check fails, the job remains in review and the audit explains why.
